Phase 07 ships a one-shot test invocation only (pass = `Ok`, fail = `Error`); the spec calls out reuse of the existing fixer cycle.
- [ ] Tighten `tests/grind_smoke.rs::drain_after_session_two_skips_session_three` so it deterministically asserts session 3 never starts. The current test polls `SessionLog::records()` every 5ms and trips drain after seeing 2 records, but the runner can race ahead into session 3 within the polling window — the assertion is relaxed to `2..=3`. The phase 07 acceptance criterion calls for the strict version. Fix needs a between-session sync point in `GrindRunner` (e.g., an async barrier the test can await) or a blocking `MockAgent` that gates per-session entry on a channel.
- [ ] Add a Ctrl-C-during-dispatch path test for `ExitCode::Aborted` (`tests/grind_exit_codes.rs::aborted_when_shutdown_abort_fires` covers only the pre-dispatch trip, by abort-before-run; the in-flight cancellation path requires either the gated mock from the drain-test deferred item or an in-process barrier).
- [ ] Close the duplicate-session-seq window in the grind resume path. `GrindRunner::run` (src/grind/run.rs:357-376) appends a session record to `sessions.jsonl` first and writes `state.json` second, with a comment claiming the order is safe. It is not: if the host process dies between the JSONL append and the state.json write, `state.last_session_seq` is stale by one, `next_seq = stale+1` collides with the already-persisted seq, and on resume the runner re-dispatches the same prompt and appends a second JSONL line with the same seq (the log has no dedup, see `SessionLog::append`). The same is true for any session whose `state.json` write failed and only logged a warn. Fix options: (a) on resume, recompute `last_session_seq` from the actual `sessions.jsonl` tail and refuse if it disagrees with the cached `state.json`, (b) write `state.json` *before* the JSONL append (with its own trade-off of advancing the scheduler past a session whose record never lands), or (c) treat `sessions.jsonl` as fully authoritative and rebuild scheduler/budget state from it on resume rather than caching them in `state.json`. Each is more than a 30-line change.
- [ ] Hooks fire on `SessionStatus::Aborted` sessions too (`post_session` always, `on_failure` because Aborted is non-Ok). On a Ctrl-C-Ctrl-C abort the user is already trying to exit, but the runner still blocks on the hook(s) until they complete or hit `hook_timeout_secs` (default 60s). Decide whether to (a) skip post_session/on_failure entirely on Aborted, (b) honor the abort token inside `run_hook` so a second Ctrl-C kills the hook child immediately, or (c) document the current behavior and leave it. Currently no test covers the abort-while-hook-running window.
- [ ] `pitboss grind` CI flake risk on `tests/grind_parallel.rs::parallel_wall_clock_is_meaningfully_less_than_sum_of_session_times`. The threshold is `0.75 * sum(sleeps)`, with sleeps tuned to 1.2s × 2 so the per-session git overhead (worktree create + ff-merge + cleanup, ~400ms on a local laptop) does not dominate. On a cold or heavily-loaded CI host the overhead can climb past the safety margin and the test will fail intermittently. Two reasonable directions: (a) measure overhead via a sequential baseline run inside the same test and assert `parallel < baseline / 1.5`, (b) replace the wall-clock assertion with an instrumentation-driven one (e.g., assert `journal.max() == 2` plus `(end - start) < some_overhead-aware bound`).
- [ ] Parallel-session verify cycles rebuild `target/` (and equivalent caches for non-cargo runners) per worktree because git worktrees do not share the main workspace's `target/`. With `verify: true` on a parallel-safe prompt and a real cargo project, every parallel session pays a full rebuild cost. Options: (a) configure `CARGO_TARGET_DIR` to a shared path passed via the agent env, (b) build once in the main workspace and skip per-session verify when the agent's commit produced a strict subset of changes, (c) document and leave it.
- [ ] The `parallel_safe` violation detection in phase 11 fires at the **worktree-sync** ff-merge (when bringing the session branch up to the current run-branch tip would overwrite the agent's uncommitted edits), not at the **run-branch** ff-merge that the spec text describes. Both catch the same property — whether the session's commits can be linearized into the run branch — but the spec language ("non-fast-forward result is a hard fail labeling the prompt's `parallel_safe: true` claim as violated") suggests the second site as the canonical check. Decide whether to (a) update the spec / docs to match the implementation, (b) drop the worktree-sync step and let the run-branch ff-merge be the only gate (which means losing the "both commits land" property when sessions touch disjoint files), or (c) keep both checks and label them distinctly in the violation summary.
- [ ] Phase 11 spec listed `SessionWorktree::merge_into(run_branch)` as a deliverable method on `SessionWorktree`, but the implementation inlines the four-step sync → commit → ff-merge → stash dance directly in `run_session_task` (`src/grind/run.rs:923-1018`). The runner is the only caller, so inlining works, but the API drift means the merge contract isn't testable in isolation and any future caller (a fixer cycle, a forensics tool, an alternative scheduler) has to re-implement it. Refactor to extract the dance into `SessionWorktree::merge_into(repo_git, run_branch, run_branch_lock, status, prompt_name) -> MergeOutcome` (or similar) and have `run_session_task` consume the resulting `MergeOutcome { commit, status, summary, sync_ok }`. Likely a 60-100 line refactor with care needed around the per-status branching that currently lives in the runner.
- [ ] Decide whether `pitboss grind --pr` failures should affect the process exit code. Phase 12 mirrors `pitboss play --pr`: a failed `gh pr create` is logged to stderr but the run's underlying exit code (Success / MixedFailures / etc.) is preserved on the theory that the work shipped successfully even if the PR step did not. That's reasonable for a transient gh outage, but it also means a CI script wrapping `pitboss grind --pr` cannot tell whether the PR was actually opened from the exit code alone. Options: (a) leave as-is and document, (b) introduce a `PrCreationFailed = 6` exit code that overrides Success when --pr was requested and the call failed, (c) add a `--require-pr` flag that opts into the strict behavior.
